Understanding Locality Sensitive Hashing(LSH): A Powerful Technique for Similarity Search.

Sarthak Joshi

·

6 min read
·
Jul 30, 2023

With recent rise of Large language models (LLMs) there has been a rise of use cases for vector databases which provide fast approximate nearest neighbour search for embeddings. Locality Sensitive Hashing (LSH) lies in the core of these vector databases and how they perform indexing.

In this article we will deep dive into workings of LSH and how it is applied within vector databases to make LLMs more powerful.

Challenge of similarity search :

When dealing with large datasets, finding similar items can be computationally expensive, especially for high-dimensional data. Brute-force approaches, which involve comparing each data point with every other point, have quadratic time complexity, making them infeasible for big data scenarios. The goal is to devise a method that can quickly identify similar items without having to exhaustively compare each pair.

Introduction to Locality Sensitive Hashing (LSH):

Locality Sensitive Hashing (LSH) is a technique that efficiently approximates similarity search by reducing the dimensionality of data while preserving local distances between points. The core idea behind LSH is to hash data points in such a way that similar items are mapped to the same hash buckets with high probability. By doing so, we can drastically reduce the search space and expedite the process of finding similar items.

How LSH Works

The LSH algorithm comprises three main steps:

Step 1: Hashing LSH involves creating multiple hash functions, each generating a hash code for every data point. These hash codes typically consist of binary strings of equal length.

Step 2: Binning The hash codes are used to distribute the data points into hash buckets. Points with the same hash code are grouped together in the same bucket.

Step 3: Querying When a query item is presented, it undergoes the same hashing process to generate a hash code. Then, we look for the bucket containing the hash code of the query item. All items within this bucket are potential candidates for being similar to the query item. We then perform a refined similarity check on these candidates using the original high-dimensional data to identify true similarities.

Key factors and parameters in LSH:

The effectiveness of LSH depends on several important parameters:

  1. Number of Hash Functions (K) More hash functions lead to a higher probability of similar items being hashed to the same bucket. However, increasing the number of hash functions also increases computational overhead.
  2. Length of Hash Codes (L) Longer hash codes provide more accurate results but also require more space to store. The trade-off between length and accuracy needs to be considered depending on the application.
  3. Number of Hash Tables (M) Using multiple hash tables (M) allows for a more robust search, as we can look for similar items across multiple hash tables.

Locality preserving hash Functions:

The key component of LSH are the locality preserving hash function. It’s a hash function which preserves the relative locality of the data points in hash space as it was in the original space. i.e an hash function h is called locality preserving if :

dist(h(q),h(r)) > dist(h(p),h(q)) given dist(q,r)>dist(p,q).

There are many types of hash functions used depending on the use case, for explanation and to keep article easy to understand we will take the example of random projection hash.

Applying LSH with random projection hash (Example):

Random projection hash works by projecting high-dimensional vectors onto random low-dimensional subspaces, resulting in lower-dimensional hash codes that still preserve the relative distances between data points.

Let us take a small example :

Consider the following four 4-dimensional vectors:

A = [1, 4, 2, 5]

B = [2, 6, 3, 7]

C = [5, 1, 4, 2]

D = [3, 2, 5, 1]

We want to hash these vectors into 1-dimensional hash codes using two random projection vectors, R1 and R2.

Step 1: Generating the Random Projection Vectors

Let’s create two random projection vectors: R1 = [0.3, 0.5, 0.2, 0.1] R2 = [0.7, 0.1, 0.6, 0.4]

Step 2: Projecting the Vectors To obtain the hash codes

For each vector, we take the dot product of the vector with both random projection vectors:

Hash1(A) = A · R1 = [1, 4, 2, 5] · [0.3, 0.5, 0.2, 0.1] = 1 * 0.3 + 4 * 0.5 + 2 * 0.2 + 5 * 0.1 = 0.3 + 2.0 + 0.4 + 0.5 = 3.2

Hash2(A) = A · R2 = [1, 4, 2, 5] · [0.7, 0.1, 0.6, 0.4] = 1 * 0.7 + 4 * 0.1 + 2 * 0.6 + 5 * 0.4 = 0.7 + 0.4 + 1.2 + 2.0 = 4.3

Similarly, we calculate the hash codes for vectors B, C, and D:

Hash1(B) = 3.2,

Hash2(B) = 5.6

Hash1(C) = 2.6,

Hash2(C) = 3.8

Hash1(D) = 2.7,

Hash2(D) = 4.6

Step 3: Assigning Hash Codes to Buckets

Let’s set two threshold values: T1 = 4.0 and T2 = 5.0. We will assign the vectors to buckets based on their hash codes:

  • Vectors with Hash1 less than or equal to T1 are placed in Bucket 1.
  • Vectors with Hash1 greater than T1 are placed in Bucket 2.
  • Vectors with Hash2 less than or equal to T2 are placed in Bucket 1.
  • Vectors with Hash2 greater than T2 are placed in Bucket 2.

Based on the thresholds, the vectors will be assigned to buckets as follows: Bucket 1: [A, C, D] Bucket 2: [B].

Now there can be cases where hash for one bucket is less than the hash value of random projection and greater for the other bucket. Here the number of hash functions play in import role and we need to remember that LSH is not a deterministic but probabilistic technique. If there are multiple hash functions the bucket for placement of vector will be chosen on the basis of majority vote.

Applications of LSH (general)

LSH has been successfully employed in various domains, including:

1. Near-Duplicate Detection Detecting similar documents, images, or audio files in large databases.

2. Nearest Neighbour Search Finding the most similar items in recommendation systems.

3. Clustering Grouping similar data points together to discover patterns and structures.

Applications of LSH in context of vector DBs and LLMs :

LSH lies at the core of indexing for many vector databases that provide approximate neighbour search. This section doesn’t deal with LSH but how vector DBs powered by LSH make LLMs more powerful.

The most important use is that vector databases act as a knowledge base for LLMs. Suppose we have an enterprise data which an organisation doesn’t want to fine tune LLM with. Using LSH powered vector DBs we can fetch the nearest context for a prompt and then these contexts are passed as prompts to the LLM to generate the required output. This can be used specifically for the use cases like enterprise chatbots and document finding.

Another important use case is for caching previous LLM responses. With vector db if we have a similar embedding cached through the previous prompts an output is generated without invoking the LLM. If it is not cached the vector DB can save the LLM response and query embedding for further use.

Conclusion

Locality Sensitive Hashing (LSH) is a powerful technique that enables efficient similarity search in large datasets. By exploiting the properties of hash functions to map similar items to the same buckets, LSH drastically reduces the search space and computational complexity. With its wide range of applications and numerous LSH variants tailored to different similarity metrics, LSH remains a fundamental tool in the arsenal of data scientists and machine learning practitioners.

We also touched upon how vector databases powered by LSH make deployment of LLMs on large scale possible. There can be various other applications of vector DBs powering LLMs but that is a topic of another article by itself.

By incorporating Locality Sensitive Hashing into our algorithms, we can unlock the potential of big data and make similarity search feasible in scenarios where traditional methods fail. Embracing LSH empowers us to build faster and more accurate systems that harness the true potential of massive datasets.
